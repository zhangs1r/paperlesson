本课目标

你要真正掌握后面所有实践课都默认你懂的那套工作流：Linux 终端、g++、CMake、build 目录、库链接和基本调试意识。

和前文怎么接

前两节课我们一直在讲系统和数学，这一节要把“我知道 SLAM 在干嘛”变成“我能开始动手写、编、跑程序”。后面第 3 到第 14 讲的所有代码，都站在这节课的基础上。

原书在这里讲得很实在：为什么要用 Linux？不是因为它“更酷”，而是因为后面会用到的大量视觉 SLAM 相关库，在 Linux 下配置和协作都最自然。你可以把它理解成：这不是课程的主题，但它是课程的地板。

为什么不用 Windows 作为默认环境

因为很多库在 Windows 下也能用，但配置成本和踩坑成本显著更高，不适合这本书的教学主线。

为什么不展开讲安装系统

因为这本书的重点是学 SLAM，不是写操作系统安装手册。安装资料网上很多，这里只抓和后面实验直接相关的部分。

1. Hello SLAM：第一个程序练的其实不是“输出字符串”

这节最表面的任务当然是写一个最简单的 C++ 程序：

#include <iostream>
using namespace std;

int main(int argc, char** argv) {
  cout << "Hello SLAM!" << endl;
  return 0;
}

但真正该练的是这一整套动作链：

$$ \text{写源文件} \rightarrow \text{调用编译器} \rightarrow \text{得到可执行文件} \rightarrow \text{运行验证} $$

书里先让你直接用 g++ helloSLAM.cpp 编译，是为了让你看到最原始的关系：C++ 源文件不是程序本身，编译后得到的可执行文件才是。

这一步和后面哪一讲会反复相遇？

几乎每一讲。第 3 讲的 Eigen、第 5 讲的 OpenCV、第 6 讲的 Ceres / g2o、第 7 讲以后的所有实验，都默认你已经知道“写一个源文件、编译它、运行它”是怎样一回事。

2. 为什么很快就要从 g++ 过渡到 CMake

书里在这里做了一个很重要的教学切换：先让你感受到“单文件直接编译”有多简单，再告诉你“工程一旦变大，这种方式马上会失控”。

于是 CMake 出场。最小版本的 CMakeLists.txt 就是：

cmake_minimum_required(VERSION 2.8)
project(HelloSLAM)
add_executable(helloSLAM helloSLAM.cpp)

它其实在做三件事：

  • 声明这是一个 CMake 工程
  • 告诉系统这个工程叫什么
  • 告诉系统要从哪个源文件生成哪个可执行目标

更关键的是，书里在这里就引入了一个后面非常常见的习惯：单独建 build 目录。

mkdir build
cd build
cmake ..
make

这个习惯的重要性远大于表面看上去的“整洁”。它意味着：源代码目录和编译产物目录分离。以后你改代码、换参数、换依赖库、重新编译，都会轻松很多。

3. 使用库：从“一个程序”走向“一个工程”

当书里开始讲库时，它其实是在帮你建立更重要的工程意识：不是所有代码都应该塞进 main 里。真正的 SLAM 工程一定会拆成函数、类、模块、库。

书里的例子非常小，但思想很正宗：先把函数声明放进头文件，再把实现写进源文件，最后通过 CMake 把它编译成库并链接到主程序。

// libHelloSLAM.h
void printHello();

// libHelloSLAM.cpp
void printHello() {
  cout << "Hello SLAM" << endl;
}

然后在 CMake 里：

add_library(hello_shared SHARED libHelloSLAM.cpp)
add_executable(useHello useHello.cpp)
target_link_libraries(useHello hello_shared)

这组动作值得你后面反复回想。因为从第 5 讲开始，你不再只是在“写一个程序”，而是在不断把 OpenCV、Eigen、PCL、Ceres、g2o 这些外部库接到自己工程里。

头文件做什么

告诉使用者：库里有哪些函数、类、接口可以调用。

源文件做什么

真正提供实现逻辑，被编译器编成目标文件和库文件。

链接做什么

让你的主程序在编译 / 运行时知道这些实现究竟来自哪一个库。

4. IDE 与调试：为什么这一讲要提 KDevelop

书里这里提 KDevelop，不是因为它必须用，而是因为作者想尽早把你拉出“只会编辑纯文本”的状态，进入稍微像样一点的工程开发体验：跳转、补全、编译、断点调试。

但这节课真正想传达的，是一个更底层的态度：无论 IDE 用什么，底层仍然是 CMake + 编译器 + 调试器。

所以你完全可以把这部分迁移成自己的现代工作流理解：

  • 编辑器 / IDE 只是界面
  • CMake 才是工程描述
  • 编译器和调试器才真正执行你的程序

这节课真正学到的“可迁移能力”

不是会不会 KDevelop，而是你知道：以后碰到任何一个第三方库工程，只要它是 CMake 工程，你都知道该怎么看它、怎么编译它、怎么把它接进自己的程序里。

5. 本课小结

Hello SLAM 真正练的是什么

不是输出一行字，而是“写源文件 → 编译 → 运行 → 验证”这条最基本开发闭环。

CMake 为什么重要

因为从这节以后，你面对的将不是单个文件，而是越来越像样的 C++ 工程。

库和链接为什么要现在学

因为后面几乎所有 SLAM 实验都会依赖外部库，而不是单文件独立程序。

课后练习

  2. 如果你忘了在 CMake 中把主程序链接到某个库上，最典型会发生什么？
    展开查看答案
    通常会在链接阶段报错，提示某些符号 / 函数找不到定义。因为头文件只告诉了编译器“有这个函数”，真正的实现还得靠链接器从对应库里找出来。
