本课目标

把 ORB 的提取、描述、匹配、筛选串成一个可执行的前端小流程，并理解每一步在后面位姿估计里的作用。

和前后文的联系

前一课给了 ORB 的原理，这一课把它落成 OpenCV 代码；下一课开始的对极几何、PnP、ICP，都默认你已经有了一批相对靠谱的匹配点对。

原书给这节实践准备了两张来自公开数据集的图像。它们的相机运动并不剧烈，正好适合我们观察：ORB 能提到什么点，匹配之后又能留下多少对可用对应关系。

从教学角度讲，这节实验其实在做五件连续的小事：

读取图像 → 检测 ORB 关键点 → 计算 BRIEF 描述子 → 按汉明距离匹配 → 过滤掉明显不靠谱的匹配

你可以把它看成第九讲 VO 前端主循环的一次极小型预演。🙂

1. 代码不是一大坨，而是五个清晰步骤

书里的示例代码虽然不长，但非常适合做“逐段带读”。如果你只是机械跑通，会觉得它像 OpenCV API 清单；如果你把每一步的角色看清楚，后面第九讲读源码时会轻松很多。

1.1 第一步：初始化 ORB

#include <opencv2/core/core.hpp>
#include <opencv2/features2d/features2d.hpp>
#include <opencv2/highgui/highgui.hpp>

Ptr<ORB> orb = ORB::create(
    500, 1.2f, 8, 31, 0, 2,
    ORB::HARRIS_SCORE, 31, 20
);

这里最值得你记住的不是参数本身，而是它们背后的前端思想：

  • 500 控制最多保留多少个关键点，对应上一课的“数量可控”。
  • 1.2f, 8 对应图像金字塔的缩放因子和层数，也就是尺度不变性的工程实现。
  • ORB::HARRIS_SCORE 说明最终保留哪些 FAST 点，不是凭随机，而是按更稳定的响应值排序。

1.2 第二步：只找位置，不算外观

vector<KeyPoint> keypoints_1, keypoints_2;
orb->detect(img_1, keypoints_1);
orb->detect(img_2, keypoints_2);

这一步只是在回答“图像里哪些位置值得记住”。它输出的是关键点位置、尺度、方向等信息，还没有真正生成描述子。

1.3 第三步：根据关键点去编码局部外观

Mat descriptors_1, descriptors_2;
orb->compute(img_1, keypoints_1, descriptors_1);
orb->compute(img_2, keypoints_2, descriptors_2);

这一步才真正产生 BRIEF 风格的二进制描述子。也就是说，detect() 和 compute() 在语义上是分开的：前者找位置，后者给这些位置贴上“外观编码”。

1.4 第四步：暴力匹配，但距离度量要选对

vector<DMatch> matches;
BFMatcher matcher(NORM_HAMMING);
matcher.match(descriptors_1, descriptors_2, matches);

这里必须是 NORM_HAMMING，因为 ORB 描述子是二进制串。上一课说过，二进制串之间最自然的相似度就是汉明距离，也就是“不同位数的个数”。

1.5 第五步：筛掉看起来就很可疑的匹配

double min_dist = 10000, max_dist = 0;
for (int i = 0; i < descriptors_1.rows; i++) {
    double dist = matches[i].distance;
    if (dist < min_dist) min_dist = dist;
    if (dist > max_dist) max_dist = dist;
}

vector<DMatch> good_matches;
for (int i = 0; i < descriptors_1.rows; i++) {
    if (matches[i].distance <= max(2 * min_dist, 30.0)) {
        good_matches.push_back(matches[i]);
    }
}

这一段看起来像经验规则，但工程上非常常见。它表达的是：先找出本次匹配里“最像”的那批点，再用它作为阈值基线。不过原书也很诚实，这不是严格理论推导，而是好用的经验式。

2. 为什么筛选规则写成 max(2 * min_dist, 30.0)

这段代码如果只是背下来，后面你就很难自己调参数。我们把它拆成两半看：

$$ \text{保留匹配} \iff \text{distance} \le \max(2\cdot \text{min\_dist},\,30.0). $$

为什么有 2 × min_dist

它想表达“只要距离没有比当前最好匹配差太多，就先留下来当候选”。

为什么还要有 30.0

因为有时最小距离会异常地小，比如接近 0。如果只按 2 × min_dist，阈值会过分苛刻，反而把很多本来还不错的匹配全丢掉。

所以这条规则本质上是在做一个平衡：既别把明显很差的匹配放进来，也别因为一两个“神仙匹配”把阈值压得过低。

3. 误匹配为什么始终是视觉前端的瓶颈

原书在这节里有一句很值得记住的话：误匹配是长期以来一直没有被彻底解决的问题。这不是夸张。

原因很现实：

  • 重复纹理太多，比如键盘、窗格、地砖、书架。
  • 局部特征只看一小块区域，看不见更大尺度的语义信息。
  • 光照变化、反光、轻微模糊都会让描述子产生漂移。

所以实际系统里的策略从来不是“匹配一次就信了”，而是层层筛：

描述子距离筛选 → 交叉检查或最近邻比值 → RANSAC / 几何一致性 → 后端鲁棒核

这也正是第六讲优化为什么重要的另一个原因：好的前端匹配能给后端一个更可靠的初值，而好的后端又能反过来抑制少量剩余误匹配的破坏力。

4. 这一节怎样通向后面的 2D-2D、3D-2D、3D-3D

当我们拿到 good_matches 之后，后面的岔路其实非常清楚：

单目：2D-2D

如果只有两帧像素坐标，就走对极几何，估计本质矩阵和相对位姿。

有一侧深度：3D-2D

如果上一帧地图点或深度已知，就走 PnP，再配合 BA 优化。

两侧都有深度：3D-3D

如果两帧都能提供三维点，就可以用 ICP 或其变种直接对齐。

所以这一课不是一节孤立的 OpenCV 入门，而是把第七讲后半段的三条求解路线同时点亮了。到第九讲你会看到，VO 前端的状态机其实就是在这些数据结构和估计器之间来回切换。🚀

课后练习

  1. 为什么 ORB 匹配中常用汉明距离，而不是欧氏距离？
    展开查看答案
    因为 ORB 的描述子是二进制串，汉明距离正好衡量不同位数的个数，计算也能通过位运算高效完成；欧氏距离更适合浮点型描述子，如 SIFT。
  2. detect() 和 compute() 在语义上分别负责什么？
    展开查看答案
    detect() 负责找关键点位置、方向和尺度等信息；compute() 负责根据这些关键点位置提取局部描述子。
  3. 为什么只靠距离筛选还不够，后面通常还要加 RANSAC？
    展开查看答案
    因为局部外观相似不等于几何上自洽。RANSAC 会进一步利用极几何、PnP 或其他模型约束，删除那些虽然描述子相似、但在全局几何上讲不通的匹配。
  4. 如果你打算把这段代码嵌入第九讲 VO 主循环，最需要额外关注什么？
    展开查看答案
    要关注实时性、特征分布是否均匀、误匹配比例、丢帧时如何回退，以及匹配结果怎样稳定地喂给 PnP / BA / 局部地图模块。
