第六讲 · g2o 实践曲线拟合

本课目标

学会从“图建模”的角度组织一个优化问题：顶点是什么、边是什么、误差在哪里算、增量怎么更新。

和前文怎么接

上一节我们刚从理论上引入图优化，这一节马上用同一个曲线拟合例子把那套语言变成 g2o 代码。这样你才能看出“图优化”到底落在了哪些类和函数上。

书里先把这个曲线拟合问题重新解释成图：

这张图特别适合作为“图优化入门样板”来看：

  • 顶点只有一个，就是参数 $[a,b,c]$；
  • 每个观测残差都是一条边；
  • 由于每条边只依赖这一组参数，所以它是一元边。

书里还顺手提到“超边 / 超图”的说法。你现在可以记住：g2o 的框架本来就是为“误差可能同时依赖多个变量”的情形准备的，这在 SLAM 里尤其常见。

1. 顶点类和边类：g2o 为什么要你手写这些

1.1 顶点类：参数长什么样，怎么更新

class CurveFittingVertex
  : public g2o::BaseVertex<3, Eigen::Vector3d> {
public:
  virtual void setToOriginImpl() {
    _estimate << 0, 0, 0;
  }

  virtual void oplusImpl(const double* update) {
    _estimate += Eigen::Vector3d(update);
  }
};

这一段最值钱的地方不是模板参数，而是 oplusImpl。书里专门停下来解释了：曲线拟合这里之所以只是简单加法，是因为参数生活在普通向量空间里。

这句解释非常重要

如果优化变量不是普通向量，而是相机位姿，那么更新就不能再是普通加法了。根据第四讲，你可能要做左乘更新或右乘更新，背后其实就是李代数扰动加到李群位姿上。

这就是为什么这节看似在讲曲线拟合，实际上已经悄悄把你带向第十一讲位姿图优化了：g2o 把“如何更新变量”这件事的控制权交给你。

1.2 边类：误差在哪儿算

class CurveFittingEdge
  : public g2o::BaseUnaryEdge<1, double, CurveFittingVertex> {
public:
  CurveFittingEdge(double x) : _x(x) {}

  void computeError() {
    const CurveFittingVertex* v =
      static_cast<const CurveFittingVertex*>(_vertices[0]);
    const Eigen::Vector3d abc = v->estimate();
    _error(0,0) = _measurement -
      std::exp(abc(0,0)*_x*_x + abc(1,0)*_x + abc(2,0));
  }
};

这段代码的核心只有一句：给定当前顶点估计值，算出当前观测残差。

你可以把它和上一节 Ceres 的 Functor 对照来看：

Ceres 风格

直接定义一个残差函数对象，然后把残差块交给 Problem。

g2o 风格

先显式定义“顶点类型”和“边类型”，再把整个问题作为图搭出来。

本质

两者都在表达同一件事，只是 g2o 更强调结构和图建模。

2. 图怎么搭起来，求解器又怎么配

书里的主程序核心结构很典型：

typedef g2o::BlockSolver<g2o::BlockSolverTraits<3,1>> Block;
Block::LinearSolverType* linearSolver =
  new g2o::LinearSolverDense<Block::PoseMatrixType>();
Block* solver_ptr = new Block(linearSolver);
g2o::OptimizationAlgorithmLevenberg* solver =
  new g2o::OptimizationAlgorithmLevenberg(solver_ptr);

g2o::SparseOptimizer optimizer;
optimizer.setAlgorithm(solver);
optimizer.setVerbose(true);

这里你不必现在就把所有模板参数都背下来，但要看出这几层分工：

  • BlockSolverTraits<3,1> 说的是：顶点维度 3，边残差维度 1。
  • LinearSolverDense 负责线性方程那一层怎么解。
  • OptimizationAlgorithmLevenberg 决定非线性迭代层用的是 L-M。
  • SparseOptimizer 则是整张图的容器。

然后是建图过程：

CurveFittingVertex* v = new CurveFittingVertex();
v->setEstimate(Eigen::Vector3d(0,0,0));
v->setId(0);
optimizer.addVertex(v);

for (int i = 0; i < N; i++) {
  CurveFittingEdge* edge = new CurveFittingEdge(x_data[i]);
  edge->setId(i);
  edge->setVertex(0, v);
  edge->setMeasurement(y_data[i]);
  edge->setInformation(Eigen::Matrix<double,1,1>::Identity() * 1/(w_sigma*w_sigma));
  optimizer.addEdge(edge);
}

这一段特别适合和第六讲开头的概率视角连起来看：setInformation(...) 本质上就是在把协方差的逆，也就是信息矩阵，交给误差边，告诉求解器“这个观测约束有多可信”。

2.1 最后优化时到底发生了什么

optimizer.initializeOptimization();
optimizer.optimize(100);

看起来只有两行，但背后其实是前面整节理论在起作用：

当前图结构 → 当前边残差与雅可比 → 线性增量方程 → L-M 更新 → 重复迭代直到收敛

也就是说，g2o 只是把你前面学过的东西模块化、工程化了。

3. g2o 和 Ceres：这节练习最想让你体会的差别

书里给出的结果大约是：

$$ \text{estimated model: } 0.890911\;\; 2.1719\;\; 0.943629 $$

和上一节 Ceres 的结果非常接近。这一点本身不意外，因为本质上还是同一个最小二乘问题。真正值得你抓住的是它们的“气质差别”：

Ceres 更像

“我把残差函数写给你，求导和迭代细节大多由库负责”。

g2o 更像

“我先把问题搭成图，再显式告诉你每类顶点、边和求解器是什么”。

所以在 SLAM 里

当问题天然就是图结构时，g2o 往往会显得非常顺手，尤其到了位姿图和 BA 场景。

书里还特别提到一点：在这两个小程序中，都没有手推雅可比，而是借助数值求导或自动方式把事情先跑起来。这对教学很友好，但它也在暗示你：越往后走，越复杂的位姿和投影问题，越值得你自己理解并掌握解析雅可比。

互动判断

这一节 g2o 曲线拟合最想训练你的，不是哪个类名，而是什么能力？

4. 本课小结

图建模真正落地了

你第一次在代码里明确看到：顶点保存变量、边计算误差、求解器组织优化。

oplusImpl 很关键

它告诉你变量如何被增量更新，这在位姿优化里会直接连接第四讲的李代数更新。

为后文埋的线

第十一讲位姿图优化基本就是这一节思路的“大型正式版”。

课后练习

  1. 为什么 g2o 里必须显式定义顶点类和边类，而 Ceres 可以只定义残差函数？
    展开查看答案
    因为 g2o 的核心心智模型是“图”，它要显式知道变量节点类型、误差边类型以及更新方式；而 Ceres 更偏向通用最小二乘接口，重点是残差块与参数块的组合。
  2. oplusImpl 为什么在曲线拟合里看起来平平无奇，但在 SLAM 里却很关键？
    展开查看答案
    因为曲线参数生活在普通向量空间里，加法就够了；但位姿生活在李群上，更新必须遵守流形结构，所以需要自己定义增量如何作用到当前估计上。
  3. 这一节和第十一讲位姿图优化的最近联系是什么？
    展开查看答案
    第十一讲的位姿图优化仍然是“顶点 + 边 + 求解器”的框架，只不过顶点变成位姿，边变成相对位姿约束，oplusImpl 也不再是普通加法，而是李代数更新。
