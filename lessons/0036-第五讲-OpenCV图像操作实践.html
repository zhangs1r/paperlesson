原书使用的是 OpenCV 3 系列，并建议从源码安装。今天实际环境未必和书里完全一样，但从学习路径上，这个建议仍然合理：你会更清楚库文件、头文件和 CMake 的关系，而不只是“装完能跑”。

# 典型流程（以书中的 OpenCV 3 为例）
sudo apt-get install build-essential libgtk2.0-dev libvtk5-dev libjpeg-dev \
  libtiff4-dev libjasper-dev libopenexr-dev libtbb-dev

unzip opencv-3.1.0.zip
cd opencv-3.1.0
mkdir build
cd build
cmake ..
make -j4
sudo make install

接着在 CMake 里找到 OpenCV：

cmake_minimum_required(VERSION 2.8)
project(imageBasics)

set(CMAKE_CXX_FLAGS "-std=c++11")
find_package(OpenCV REQUIRED)
include_directories(${OpenCV_INCLUDE_DIRS})

add_executable(imageBasics imageBasics.cpp)
target_link_libraries(imageBasics ${OpenCV_LIBS})

这段 CMake 真正在做什么

find_package 找库，include_directories 让编译器知道头文件在哪，target_link_libraries 让链接器知道最终可执行文件要把哪些库连进来。以后看别的 C++ 视觉工程，也几乎都逃不开这三步。

1. 第一段真正的图像程序：读进来、看见它、确认它是什么

书里这一段代码非常适合当“图像程序最小闭环”：读图、判空、打印尺寸、显示、阻塞等待。

cv::Mat image = cv::imread(argv[1]);
if (image.data == nullptr) {
    cerr << "文件不存在." << endl;
    return 0;
}

cout << "图像宽为 " << image.cols
     << "，高为 " << image.rows
     << "，通道数为 " << image.channels() << endl;

cv::imshow("image", image);
cv::waitKey(0);

这里每一步都不该被跳过：

  • imread 把磁盘上的图像文件读成 cv::Mat
  • image.data == nullptr 是基本的健壮性检查
  • rows / cols / channels() 让你先确认自己手里拿到的到底是什么图
  • imshow + waitKey 用最直接方式确认程序确实读到了你以为的图像

这和前面的理论不是两张皮

你打印出来的宽、高、通道数，就是上一节里讲的图像结构；你此时手里看到的像素网格，就是前面几节几何模型最终投到程序里的载体。

2. 逐像素遍历：真正理解 ptr<> 为什么快

书里推荐用行指针遍历图像。这不是炫技，而是因为它非常贴近底层内存布局，速度也往往比高层逐点访问更好。

for (size_t y = 0; y < image.rows; y++) {
    unsigned char* row_ptr = image.ptr<unsigned char>(y);
    for (size_t x = 0; x < image.cols; x++) {
        unsigned char* data_ptr = &row_ptr[x * image.channels()];
        for (int c = 0; c != image.channels(); c++) {
            unsigned char data = data_ptr[c];
        }
    }
}

这里建议你脑中同步想象一张三通道彩色图的内存排布：同一行里，像素一个接一个铺开，而每个像素内部又是 B、G、R 连着存。所以走到第 x 个像素时，要乘上通道数才对。

为什么先拿行指针

因为每次进入新的一行，只需定位一次行起点，后面列内偏移就很自然。

为什么再乘通道数

因为一个像素可能不止一个字节。彩色图走一步，不是走 1，而是走 3。

为什么书还加了计时

因为在视觉程序里，很多算法是否可实时，往往就卡在这种看似平常的像素遍历上。

自查一下

如果你把 x * image.channels() 错写成了 x，彩色图里会发生什么？

3. 深拷贝 vs 浅拷贝：为什么一改局部，原图也会跟着变

很多同学第一次遇到 cv::Mat，最惊讶的不是 API，而是赋值后的行为。书里专门讲这个，特别好，因为它直接关系到你后面调试时会不会怀疑人生。

cv::Mat image_another = image;      // 浅拷贝，共享底层数据
cv::Mat image_clone = image.clone(); // 深拷贝，复制底层数据

浅拷贝意味着：两个对象头部是不同的，但指向同一块像素内存。所以你改其中一张图像的局部，另一张也会“同步受伤”。而 clone() 或 copyTo() 才会真正复制底层数据。

操作是否共享底层像素适用场景
Mat B = A共享想省拷贝、只读或明确知道共享语义时
A.clone()不共享想独立修改一份副本时
A.copyTo(B)不共享需要拷贝到已有对象时

为什么这和后面点云拼接也有关

第 5.4 节会同时处理彩色图、深度图和位姿。如果你在工程里把某张图“顺手赋值后又修改”，可能会不小心破坏原始观测数据，后面定位和建图就会一起受影响。

4. 本课小结

工程入口

会写 find_package 和 target_link_libraries，就算真正开始会搭图像工程了。

像素访问

ptr<> 的本质是按行拿到底层指针，再按列和通道做偏移。

拷贝语义

cv::Mat 默认赋值是浅拷贝，真正要独立副本时得显式深拷贝。

课后练习

  1. 为什么书里不建议一上来就只用最省事的高层 API，而是还要讲 ptr<> 和底层访问？
    展开查看答案
    因为视觉程序常常需要高效的逐像素处理，理解底层布局后，你才能在性能、正确性和调试上更有把握，而不是只会调用接口却不知道它在做什么。
  2. 为什么 Mat B = A 后修改 B 可能会影响 A？
    展开查看答案
    因为这是浅拷贝，两个 cv::Mat 对象共享同一块底层像素数据。要想彼此独立，必须使用 clone() 或 copyTo()。
  3. 下一节拼接点云时，为什么这一节的“像素读取 + 通道顺序 + 深度类型”都会再次出现？
    展开查看答案
    因为点云拼接需要逐像素读取颜色图和深度图：深度图给出几何距离，彩色图给出点云颜色。如果你不清楚类型、通道和索引顺序，点的位置和颜色都会读错。
